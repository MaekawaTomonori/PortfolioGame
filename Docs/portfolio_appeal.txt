# 1. 資料全体コンセプト
────────────────────────────────────────────────────────────────────────────────

■ この資料の主題
  「自作ゲームエンジン（静的ライブラリ）とゲーム実装の完全分離設計」を軸に、
  C++ / DirectX 12 による本格的なリアルタイム 3D ゲームシステムの
  設計力・実装力・品質管理意識をアピールする。

■ 読み手に持ち帰らせるべき印象
  - 「Engine と Game の依存方向を徹底管理しており、再利用可能な設計ができる人材だ」
  - 「DirectX 12 のリソース管理・描画パイプラインを自前で組み立てられる人材だ」
  - 「Repository / Loader / Handle のような設計パターンを適切に使える人材だ」
  - 「デバッグ支援や品質向上の小さな工夫を積み重ねられる人材だ」

■ 構成の軸
  ① アーキテクチャ（Engine–Game 分離・依存方向管理）
  ② DirectX 12 描画パイプライン（自作実装）
  ③ リソース管理設計（Repository / Loader / TextureManager）
  ④ ポストエフェクト・パーティクル等の拡張実装
  ⑤ デバッグ支援の充実度

# 4. before / after で見せるべき改善点
────────────────────────────────────────────────────────────────────────────────

【改善例 ①: リソース生成の責務分離】
  改善前: Game 側が描画オブジェクト（Model 等）を直接生成・保持していた
  改善後: Loader → Repository → Handle を経由し、Game はハンドル経由でのみアクセス
  何が良くなったか:
    - Game が描画内部（GPU リソース）を知らなくて済む構造になった
    - 同一リソースの重複ロードがなくなり、メモリ効率が向上した
    - ロード方式（glTF / OBJ）の切り替えが Game 側に影響しない
  プログラマとしての強み: 責務分離・カプセル化・パターン適用の判断力

【改善例 ②: 描画タスクの分類管理】
  改善前: シーンの Draw() がスワップチェーンに直接書き込んでいた
  改善後: Renderer のタスクキュー（pp_ / tasks_ / uiTasks_）に登録し後段で一括処理
  何が良くなったか:
    - ポストエフェクトの適用有無をシーン側が意識せずに指定できる
    - UI（ImGui）を常にスワップチェーンに描画し、エフェクトによる歪みを防いだ
    - 描画順の制御が一箇所に集約され保守性が向上した
  プログラマとしての強み: パイプライン設計・関心の分離・拡張性の設計

【改善例 ③: マジックナンバーの定数化（現ブランチの対応）】
  改善前: ゲームロジック中に数値リテラルが散在していた（速度値・HP 値等）
  改善後: 名前付き定数・設定ファイル外出し等による管理
  何が良くなったか:
    - 数値の意味が名前から自明になりコードの可読性が向上した
    - 調整が一箇所の変更で完結し、バグの混入リスクが減少した
  プログラマとしての強み: コード品質意識・保守性への配慮・リファクタリング実施力

【P02: アーキテクチャ全体図（Engine–Game 分離設計）】

  見出し案 : 「依存方向を一方向に制御した Engine–Game 分離アーキテクチャ」

  一番伝えたいこと:
    Game は Engine を利用するが、Engine は Game を一切知らない。
    この依存方向の徹底管理が再利用性・保守性の基盤となっている。

  本文ドラフト:
    本プロジェクトは、エンジン機能を提供する静的ライブラリ「Engine」と、
    ゲームロジックを担う「Game」を物理的・論理的に分離した構成を採用しています。
    依存関係は Game → Engine の一方向のみとし、Engine は Game の実装を
    一切参照しない設計としました。

    この方針により以下を実現しています。
      ・Engine を他プロジェクトへそのまま転用可能な再利用性
      ・ゲームロジックの変更が Engine コンパイルに影響しない独立性
      ・Engine API の安定性維持（内部実装変更が Game 側に露出しない）

    Engine は公開 API（include/）を通じてのみ Game からアクセス可能とし、
    DirectX 12 のオブジェクト（ID3D12Device 等）を Game 側に露出しない
    設計としています。

  図の案:
    Engine ボックス（include/ が公開面） ← Game ボックス
    Engine ボックスの内部に externals（Assimp / ImGui 等）も包含

  補足案:
    「API の境界を明確にすることで、Engine の内部改修が Game 側に
    波及しない設計を意識して実装しました」

【P04: リソース管理設計（Repository / Loader パターン）】

  見出し案 : 「Repository / Loader パターンによる 3D リソースの一元管理」

  一番伝えたいこと:
    Model / Mesh / Texture を直接生成させず、Repository 経由に統一することで
    重複ロードを防ぎ、Game 側を GPU 詳細から切り離した。

  本文ドラフト:
    本エンジンのリソース管理は、Repository / Loader パターンを採用しています。
    Model クラスは初期化時に ModelRepository にキャッシュが存在するか確認し、
    未読み込みの場合のみ GltfLoader / ObjLoader を通じてロードを実行します。

    IModelLoader インターフェースを設けることで、ファイルフォーマット（glTF / OBJ）の
    切り替えを Game 側・Repository 側に影響を与えずに実現しています。

    TextureManager も同様のキャッシュ設計を採用し、同一パスのテクスチャが
    二重に GPU へアップロードされることを防いでいます。

    この構成により:
      ・同一モデルデータの重複ロードを排除
      ・Game 側が GPU リソースの詳細を知らずに済む設計
      ・ロード方式の追加が既存コードに影響しない拡張性

  図の案:
    ResourceRepository → ModelRepository / MeshRepository / TextureManager
    IModelLoader ← GltfLoader / ObjLoader の継承関係
    Model::Initialize() → Repository.Get() → (miss) → Loader → Register のフロー図

【P03: DirectX 12 描画パイプライン構築】

  見出し案 : 「DirectX 12 を全手動構築した描画パイプライン」

  一番伝えたいこと:
    ラッパーや高水準 API に頼らず、Device / CommandList / Fence を
    自前で管理している DirectX 12 の実装力。

  本文ドラフト:
    本エンジンでは DirectX 12 を使用した描画パイプラインを自前で構築しています。
    DirectXAdapter クラスが Device / CommandAllocator / CommandList / SwapChain /
    Fence / RTV / DSV の初期化と管理を一手に担い、Game 側にはこれらを
    一切露出しない設計としています。

    GPU リソースは DX12Resource クラスでラップし、リソースの状態遷移
    （D3D12_RESOURCE_STATES）を追跡管理しています。
    これにより State Transition バリアの記述を適切に行い、
    GPU 側の同期不整合を防いでいます。

    GPU–CPU 同期には Fence を使用し、フレームごとに WaitForFenceValue を
    呼び出すことで描画完了を確認しています。

  図の案:
    BeginFrame → Reset CommandAllocator → Clear RTV/DSV →
    Scene Draw → PostEffect → UI → Close CommandList →
    ExecuteCommandLists → Present → WaitForFenceValue のフロー

  補足案:
    SRVManager により CBV/SRV/UAV ディスクリプタヒープのスロット管理も
    自前で実装しており、テクスチャ登録時に自動的にスロットを割り当てています。

# 6. 細やかな実装アピール項目一覧
────────────────────────────────────────────────────────────────────────────────

■ メモリ安全性・null 安全性
  - ModelRepository::Get() は nullptr を返却しポインタ安全アクセスを保証
  - unique_ptr による自動メモリ管理（RAII 準拠）
  - GPU マップポインタ（wd_, mappedInfluence 等）の初期値 nullptr 保証
  - optional<int32_t> による親ジョイント有無の安全な表現

■ GPU リソース管理
  - DX12Resource による GPU リソース状態追跡（State Transition バリア自動管理）
  - SRVManager によるディスクリプタスロット番号の一元割り当て
  - Fence / WaitForFenceValue による CPU–GPU 同期
  - GPU マップ済みポインタ経由による毎フレーム更新（SetTranslate 等）
  - RenderTexture 経由のポストエフェクト合成（スワップチェーン汚染回避）

■ キャッシュ・ロード効率化
  - ModelRepository / MeshRepository / TextureManager による重複ロード防止
  - unordered_map による O(1) キャッシュ参照
  - TextureManager の遅延ロード（初回参照時のみ GPU アップロード）

■ 描画秩序の制御
  - Renderer の 3 タスクキュー（pp_ / tasks_ / uiTasks_）による描画順固定
  - UI（ImGui）を常にスワップチェーンに描画しポストエフェクトの影響を排除
  - モデル描画の WVP 行列を UpdateMapData で GPU に毎フレーム自動転送

■ デバッグ支援
  - DebugUI::RegisterCommand() によるデバッグコマンドの動的登録
  - DebugUI::IsVisible() による各ウィンドウ表示状態の管理
  - DebugUI の mutex 保護によるスレッド安全なコマンド登録
  - CollisionManager::DrawDebug() による衝突形状（球・AABB）のライン可視化
  - Model::Debug() / Mesh::Debug() による Transform・アニメーション状態の ImGui 表示
  - WatchDebugger による変数リアルタイム監視
  - PerformanceProfiler による FPS 制限・表示
  - SceneView でゲーム画面を ImGui ウィンドウ内に描画（座標変換も対応）
  - Input::SetSceneViewTransform() によるゲーム座標とスクリーン座標の補正

■ 設計品質
  - CLAUDE.md による依存方向ルールの明文化（チーム・AI 向け共通ガイドライン）
  - IScene / IGame による Engine–Game 境界インターフェースの明確化
  - IPostEffect 継承によるエフェクト追加時の既存コード無変更設計
  - IModelLoader（Strategy パターン）による拡張可能なローダー設計
  - SceneFactory / PostEffectFactory によるオブジェクト生成の一元管理
  - MagicEnum による enum の文字列化（デバッグ・ログ出力の可読性向上）
  - spdlog 統合による構造化ロギング
  - nlohmann/json による設定データの外出し対応基盤

■ モダン C++ 活用
  - C++20 std::span による配列参照（コピーなしの安全なスライス）
  - std::optional によるジョイント親インデックスの表現

  - ラムダ関数によるレンダータスクのキュー登録
  - std::function<void()> によるポリモーフィックなタスク格納

# 7. 避けるべき書き方チェックリスト
────────────────────────────────────────────────────────────────────────────────

  □ 取説化していないか
    → 「操作方法はXキー」「ゲームの遊び方」等の記述を避ける

  □ プログラマ職以外のアピールになっていないか
    → ゲームデザイン・演出意図・世界観・UI デザインの見た目を主題にしない

  □ 自分以外を主役にしていないか
    → Assimp / ImGui 等のライブラリ自体を「使いました」で終わらせず、
       「どう統合・設計したか」を書く

  □ ネガティブな書き方になっていないか
    → 「未完成」「技術不足」などをストレートに書かない。
       改善点は「改善の方向性・判断」として前向きな文脈にする

  □ 1 ページ 1 テーマになっているか
    → 各ページに単一の見出しと主題を設定する

  □ ビジュアル前提になっているか
    → テキストのみで説明を完結させず、図・画面・フロー図を配置する

  □ 「だから何？」で終わらない構成になっているか
    → 各技術説明の末尾に「それが何の強みにつながるか」を明示する
